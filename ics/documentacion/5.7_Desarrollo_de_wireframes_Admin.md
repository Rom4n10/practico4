# 5.7 Desarrollo de Wireframes - Panel de Administración

## 1. Funcionalidad Seleccionada
**Módulo:** Gestión de Tipos de Evento (Dashboard Admin - `index.html`)
**Descripción:** El flujo de administración central, específicamente la visualización del listado de eventos y el proceso para crear/editar un "Nuevo Tipo de Evento" mediante un modal sobrepuesto, sin perder el contexto de la página principal.

---

## 2. Flujo Completo Representado

El diseño representa un flujo de administración (CRUD básico) que consta de las siguientes vistas y estados:

1.  **Paso 1: Vista Principal (Dashboard)**
    *   **Punto de entrada:** Pantalla inicial con el listado de tipos de evento.
    *   **Contenido:** Barra lateral de navegación (Sidebar), barra superior con perfil/tema, filtros de búsqueda, y la lista de eventos (con toggle para ver en modo Tabla o Grilla).
2.  **Paso 2: Apertura del Modal de Creación**
    *   **Acción:** El usuario hace clic en el botón primario "Nuevo Tipo de Evento".
    *   **Contenido:** Se oscurece el fondo (backdrop) y aparece un Modal superpuesto con el formulario de creación dividido en secciones lógicas (Información Básica, Duración, Modalidad, Descripción y Zona de carga de archivos).
3.  **Paso 3: Interacción y Llenado de Datos**
    *   **Contenido:** El usuario interactúa con inputs de texto, botones seleccionables (píldoras para duración), selectores visuales de modalidad (presencial/virtual) y sube imágenes arrastrando y soltando (Drag & Drop).
4.  **Paso 4: Confirmación (Guardado)**
    *   **Acción final:** Clic en "Crear Tipo de Evento" o presionar `Ctrl + S`.
5.  **Punto de Salida y Feedback**
    *   El modal se cierra y aparece una notificación tipo **Toast** (pequeño cartel flotante temporal) confirmando que el evento se creó exitosamente, y la tabla de fondo se actualiza.
6.  **Pantallas de Contingencia / Extras**
    *   **Modal de Error:** Si la acción falla, un diálogo interrumpe informando el error exacto.
    *   **Panel de Personalización:** Un panel lateral (Theme Panel) para cambiar el color, densidad o modo oscuro/claro de la interfaz.

> **Consistencia con Requerimientos:** Se cumple con mostrar un flujo administrativo completo, desde la visualización general hasta el ingreso de un nuevo registro, validando la entrada y confirmando la acción.

## 3. Aplicación de Principios de Usabilidad (8 Reglas de Oro de Shneiderman)

Para este módulo administrativo, se priorizaron principios enfocados en la eficiencia y el control del usuario experto:

### Principio A: Que los usuarios usen atajos (Regla 2)
*   **Cómo se aplica:** Al ser un panel de administración pensado para uso frecuente, se incorporaron **Atajos de Teclado (Shortcuts)**. El usuario puede presionar `Ctrl + N` para abrir el modal de nuevo evento, `Ctrl + S` para guardarlo, `Ctrl + F` para ir al buscador, o `Esc` para cancelar. Esto agiliza enormemente el trabajo repetitivo de los administradores.

### Principio B: Ofrecer un manejo simple de errores (Regla 5)
*   **Cómo se aplica:** A nivel de interfaz, si el usuario olvida un campo obligatorio (marcado con `*`), el formulario bloquea el envío y resalta visualmente el error. A nivel de sistema, si ocurre una falla inesperada (por ejemplo, una pérdida de conexión al intentar guardar), la aplicación despliega automáticamente un Modal de Error claro que le explica al usuario qué salió mal y cómo puede reintentar la acción, evitando que la pantalla quede congelada o colapse sin dar explicaciones.

### Principio C: Soporta un punto de control interno (Regla 7)
*   **Cómo se aplica:** Se le da el control total de la interfaz al usuario mediante el **Panel de Personalización**. El usuario puede decidir si prefiere trabajar en "Modo Oscuro" para no cansar la vista, cambiar la densidad de la información (más junta o espaciada) y elegir el color de acento de la aplicación. El sistema se adapta a las necesidades del usuario, no al revés.

### Principio D: Trabajar por la consistencia (Regla 1)
*   **Cómo se aplica:** Existe un "Design System" estricto. Todos los botones primarios tienen el mismo color, forma y comportamiento al pasar el mouse (hover). Todos los modales se abren con la misma animación de desvanecimiento (fade-in) y se cierran con el mismo botón en la esquina superior derecha. La iconografía (íconos de pluma, tacho de basura, etc.) mantiene el mismo estilo visual en todo el panel.

## 4. Resoluciones y Contextos de Uso

Al ser un panel de administración pesado en datos, el enfoque principal es *Desktop*, pero se adapta correctamente a pantallas menores:

*   **Versión Desktop (Principal):** 
    *   Barra lateral (Sidebar) expandida permanentemente a la izquierda.
    *   Tabla de datos con múltiples columnas visibles para análisis rápido.
    *   Modal de creación ancho y centrado en la pantalla.
*   **Versión Mobile/Tablet:** 
    *   El Sidebar se colapsa en un menú hamburguesa para ahorrar espacio.
    *   La "Vista de Tabla" se esconde automáticamente o permite scroll horizontal, dándole prioridad a la "Vista de Tarjetas (Grid)" que es más amigable para pantallas táctiles.
    *   El modal de creación ocupa el 100% del ancho y alto de la pantalla, comportándose como una página nueva para evitar que el teclado virtual tape los campos.
